# 项目开发规则

> 此文件用于存储项目特定的开发规则和偏好，帮助 AI 助手更好地为你服务。

## 文件命名规则

### 1. 序号规则

所有新生成的文件（包括程序文件、调试测试文件、接口文件等），在文件夹内都以**5位数字序号**编排：

- **起始编号**: `00001`
- **递增规则**: 每生成一个新文件，序号自动 `+1`
- **格式**: `00001 filename.ext`
- **示例**: 
  - `00001 main.py`
  - `00002 utils.py`
  - `00003 api_client.py`

### 2. 说明文件规则

每个程序文件、调试测试文件、接口文件**必须**有对应的说明文件：

- **命名格式**: `[与被说明主体相同的序号] [主体文件名].description.md`
- **关联方式**: 说明文件的序号必须与被说明的主体文件完全一致
- **示例**:
  - 程序文件: `00005 main.py`
  - 对应说明文件: `00005 main.description.md`

### 3. 独立说明文件规则

对于**没有主体文件**的说明文档（如项目总体架构、数据流程、通信规范等）：

- **编号格式**: `A` 开头 + 4位数字，从 `A0001` 开始
- **递增规则**: 每生成一个独立说明文件，序号自动 `+1`
- **示例**:
  - `A0001 项目总体架构.md`
  - `A0002 数据流程说明.md`
  - `A0003 通信规范.md`

## 规则总结

| 文件类型 | 编号格式 | 起始编号 | 示例 |
|---------|---------|---------|------|
| 程序/测试/接口文件 | 5位数字 | 00001 | `00001 main.py` |
| 对应说明文件 | 与主体相同序号 | - | `00001 main.description.md` |
| 独立说明文档 | A + 4位数字 | A0001 | `A0001 项目总体架构.md` |

## 注意事项

- 每次创建新文件前，先检查当前目录中已有的编号，确保新文件使用正确的递增序号
- 说明文件必须与被说明的程序文件保持序号一致
- 独立说明文档（A开头）和普通程序文件（纯数字）的编号系统相互独立

## Git 提交规则

### Markdown 文件提交约束

本项目对 Markdown 文件的 Git 提交有严格约束：

1. **README.md** - 唯一会被推送到 GitHub 的 Markdown 文件
   - 这是项目的门面，面向公众展示
   - 应包含项目简介、快速开始、基本使用方法

2. **其他所有 .md 文件** - 仅保留在本地，不提交到 GitHub
   - 包括所有 `*.md`、`*.description.md`、说明文档等
   - 这些文件通过 `.gitignore` 规则被排除在版本控制之外

### 本地专用文件夹

以下文件夹**仅保留在本地**，不提交到 GitHub：

| 文件夹名 | 用途 | 本地路径 |
|---------|------|---------|
| `Github Polymarket Skills/` | 参考技能文档、调研资料 | 项目根目录下 |
| `KimiSkill/` | 角色系统配置、技能文件 | 项目根目录下 |

### 技术实现

`.gitignore` 中已配置：
```gitignore
# Markdown 文档 - 只保留 README.md 在 GitHub，其他仅在本地
*.md
!README.md

# 本地专用文件夹 - 仅在本地保留，不提交到 GitHub
Github Polymarket Skills/
KimiSkill/
```

### 后果

- ✅ README.md 会同步到 GitHub，供访客浏览
- ❌ 所有其他说明文档、设计文档、架构文档仅在本地可用
- ❌ `Github Polymarket Skills/` 和 `KimiSkill/` 文件夹仅在本地可用
- ⚠️ 执行 `git add .` 时不会意外将上述文件/文件夹加入暂存区
